Headline

A typical active Space costs ≈ $16.50 / month (≈ $198 / year) on Composer 2.5 Fast. Swapping in a top open-source model cuts this 3×–24× — the cheapest credible option (DeepSeek V4-Flash) lands the typical Space at ~$0.70 / month.

Open-source model pricing

The same workload on the most capable open-weight models of 2026. Highly relevant here: Composer 2.5 is itself post-trained from Moonshot's Kimi K2.5, so running the open Kimi line directly is the closest like-for-like swap — at a fraction of Composer Fast's interactive premium.

Hosting is already included

All rates here are managed / hosted inference (per-token) — the provider runs the model on their GPUs and bills per token (Cursor for Composer; OpenRouter / DeepSeek / DeepInfra for the open models). There is no separate GPU/infra cost to add. Self-hosting the open weights is a different fixed-cost model (e.g. one H100 ≈ $1,400+/month) that is more expensive at per-Space volumes and only pays off when many Spaces share GPUs — so it is out of scope for a per-Space estimate.

Takeaways

  • The workload is input-heavy (~96% of tokens are input — large system prompt + re-sent history + tool results). Output price barely moves the total, so models with cheap input (DeepSeek) win disproportionately.
  • Kimi K2.6 is the natural "same quality, own the stack" choice given Composer's Kimi lineage, at ~3× lower cost than Composer Fast.
  • DeepSeek V4-Flash makes the AI layer almost free per Space (~$0.70/mo) — a strong default for background features where latency and peak reasoning matter least.
  • Tiered routing (premium model for interactive copilot, cheap model for background jobs) likely lands a typical Space at $1–3/month with no perceptible quality loss.

Two vote types — most votes do not execute

Non-executing vote: a member casts yes/no while the proposal is still open. It records voting power but does not run the agreement on-chain.
Deciding vote: the vote that reaches quorum/unity and auto-executes the proposal via the Executor (447k gas verified for a mint-scale agreement — execution bundled in this tx).
Every proposal has exactly 1 deciding vote and typically many more non-executing votes before it. The monthly model reflects this: e.g. Typical = 15 deciding vs 195 non-executing (13:1). Non-executing votes are modeled at 150k gas each ($0.0013/vote at 0.005 Gwei) — not yet verified on mainnet.

Gas remains negligible vs AI

With active governance, a Typical space pays $0.37/month on-chain vs $16.52/month for AI (Composer 2.5 Fast) — roughly 45× less. Non-executing votes at 150k gas each are the largest line item. Only Heavy scale (~$3.27/mo) approaches meaningful on-chain spend — still well below AI.

Caveats. L2 execution gas only at 0.005 Gwei; Base adds a small L1 data fee per tx (typically sub-cent). Gas scales linearly with gas price. Who pays (member wallets vs. paymaster) is separate from the aggregate cost shown here. Re-run gas-cost-analysis.ts on Base to capture a non-executing vote and a successful join.

Methodology & assumptions

  • In-space system prompt ≈ 5.6k tokens, re-sent on every step.
  • Tool loop capped at 6 steps; full history re-sent each step (no truncation).
  • No max_tokens cap configured → output sizes are conservative estimates.
  • Org-memory reads capped at 48k chars (~12k tokens) per fetch.
  • Orchestrator capped at 3 auto-signals/day/space; discussion refresh ~4 runs/day.
  • Costs scale ~linearly with task volume across profiles.

Levers to reduce cost (any model)

  • Trim / cache the system prompt (cached reads 60–80% cheaper on most open hosts).
  • Truncate conversation history instead of re-sending the full transcript.
  • Route background work to a cheap/standard tier.
  • Set a max_tokens cap to bound output.
  • Cap memory deep-dive context below 48k chars for routine summarization.
