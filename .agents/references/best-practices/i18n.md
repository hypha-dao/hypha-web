### i18n Best Practices

#### Do

- Add new keys to **all** locale files simultaneously — never leave a locale missing keys
- Use `getTranslations('Namespace')` from `next-intl/server` in Server Components
- Use `useTranslations('Namespace')` from `next-intl` in client components
- Use namespaced keys (PascalCase namespace, camelCase key): `t('createSpace')` not `t('Create Space')`
- Use ICU message format for dynamic values: `{count}`, `{daysLeft}` — never positional
- Put shared strings in `Common` namespace, feature-specific strings in their own namespace
- Pass the `Locale` type through component props from route params
- Validate that all locale JSON files have identical key sets
- Coordinate translation key additions with the feature PR — same branch, same review
- Use the `Locale` type from `@hypha-platform/i18n` — never use `string` for locale parameters

#### Avoid

- Using the old dictionary-based system (`get-dictionaries.ts`, `i18n-config.ts`) — use `next-intl`
- Hardcoding locale strings (`'en'`, `'de'`) — reference `routing.locales` and `routing.defaultLocale`
- Storing translations in component files — all translations belong in `packages/i18n/src/messages/`
- Creating locale-specific components — use `useTranslations` to parameterize a single component
- Using `useRouter` from `createNavigation` for locale switching — causes double-prefixing; use `window.location.href`
- Creating a fresh `NextResponse.next()` after i18n middleware — drops the `X-NEXT-INTL-LOCALE` header
- Adding `'use client'` to server components without checking they don't use server-only APIs
- Allowing message JSON keys to diverge between locales — enforce parity in PR review
