### i18n Best Practices

#### Do

- Keep dictionary keys as readable English strings — they serve as both identifier and fallback
- Add new keys to **all** locale files simultaneously — never leave a locale missing keys
- Use `getDictionary` in Server Components — it is `server-only` and must not leak into client bundles
- Pass the `Locale` type through component props from route params — avoid re-reading cookies in components
- Keep dictionary files flat — avoid deep nesting until dictionary size warrants namespacing
- Validate that all locale JSON files have identical key sets (no missing or extra keys)
- Test locale redirects: `/` -> `/en/network`, `/de` -> `/de/network`
- Verify the `HYPHA_LOCALE` cookie is set correctly after middleware runs
- Use the `Locale` type from `@hypha-platform/i18n` — never use `string` for locale parameters
- Coordinate dictionary key additions with the feature PR — same branch, same review

#### Avoid

- Importing from `@hypha-platform/i18n` (server entry) in client components — use `@hypha-platform/i18n/client` instead
- Hardcoding locale strings (`'en'`, `'de'`) — reference `i18nConfig.locales` and `i18nConfig.defaultLocale`
- Storing translations in component files — all translations belong in `packages/i18n/src/dictionaries/`
- Creating locale-specific components — use the dictionary function to parameterize a single component
- Adding translation logic to `packages/epics` or `packages/ui` — those packages consume the `Locale` type only
- Breaking the middleware chain — `i18nMiddleware` must run before CSP middleware for correct locale detection
- Using raw string concatenation for locale-prefixed URLs — use the `lang` param from route context
- Allowing dictionary JSON keys to diverge between locales — enforce parity in PR review
